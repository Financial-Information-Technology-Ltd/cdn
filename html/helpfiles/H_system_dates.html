System Dates

This screen allows the system administrator to set the various dates which affect processing on LIFEfit.

In most cases, these dates are set manually.

It is possible to include batch process LF.SET.SYSTEM.DATES (to be run as the last process in a daily batch) which sets all dates (with the exception of the commission statement date) to the next working day.

Label

Description

Key

System identifier. Read Only.

Effective Date

System effective date.

This will be used as the default effective date for all accounting transactions.

System Status

Select from dropdown.

If the status is anything other than System Available, login is only permitted for System Administrators (users with the administrator option ticked on their user profile).

The rationale is to prevent end user access until the results of the daily batch run have been checked and signed off.

Typically, once the batch has been signed off, the system date is set to the next working day and System Available is selected here.

Batch Processing

Indicator, set automatically by LIFEfit, showing that accounting records are being copied to a journal history file (only one process at a time may do this)

High Regular Income Payments Date

Date up to which regular income payments (RIP future records) will be processed.

These are regular payments by deduction of policy value, either unit-linked or deposit administration.

Some companies will only update this once a month, so all policies regular income payments are processed on the same day.

High Annuity Payments Date

Date up to which annuity payments (ANPP future records) will be processed.

Some companies will only update this once a month, so all policies annuity payments are processed on the same day.

Current commission statements date

Effective date up to which commission transactions are included in commission statements

Last Reins Finance Dt

Last reinsurance finance date (used by some companies in control of monthly reinsurance finance accounting and reporting)

Unit Run Date

Date up to which unit-related future records are processed.

This is often different from the current system date, especially for companies where unit prices are not available same day

For example, it may be kept one day behind for daily pricing where prices are calculated one day in arrears, or it may only be updated once a month if funds are priced monthly.

It prevents attempts to process unit transactions where no price is expected to be available. These do not cause errors, as future records are not processed until prices are available, but can make batch reports look a little untidy.

Mortality High Date

Date up to which triggers for mortality premium payment by unit deduction are processed (UMD future records).

Unit move from date

Start date of unit movements to be included in unit movement (LF159) report.

Unit move to date

End date of unit movements to be included in unit movement (LF159) report.

Unit Statement date

Effective date up to which transactions will be included in automatically-generated Unit Statements / Premium Statements (typically triggered on policy anniversary).

Cur Reins Finance Dt

Current Reinsurance Finance Date (used by some companies in control of monthly reinsurance finance accounting and reporting).
