Workflow Task Code

This screen is used to define the workflow codes that are used to manage work in the back office. Each code represents a category of work that takes place, and needs to be tracked.

Label

Description

Code

Workflow task code

Name

Workflow task name

Due after (hours:minutes)

Maximum time allowed for this task.

 

This is part of the RAG (Red Amber Green) rating system. RAG rating provides a visual reminder of the urgency of a piece of work in the user's "to do" list; after this time, the workflow task indicator turns red.

 

Please note that the target time "skips" non-standard hours (as defined at Admin > Config Application, Working day start and end time) and non-working days (weekends and holidays).

 

This information is often used for Service Level Agreement (SLA) reporting.

Target time after (hours:minutes)

Target processing time for this task.

 

This is again part of the RAG (Red Amber Green) rating system; after this time, the workflow task indicator turns amber (skipping non-standard working hours, weekends and public holidays).

Workflow Group

Select from the dropdown as defined in CODES record WORKFLOW_TASK_GROUP.

 

This enables “grouping” of tasks, for example by operating department. It simplifies the classification of work when the user is creating workflow manually.

 

e.g. New Business tasks, Claim Tasks, Actuarial Tasks.

 

CODES record WORKFLOW_TASK_GROUP can be maintained by your LIFEfot administrator.

 

Default User Group

When a workflow item is created automatically by a policy event, diary schedule or as part of a workflow chain, it will appear in this user group's group "to do" list.

 

Default User

When a workflow item is created automatically by a policy event, diary schedule or as part of a workflow chain, it will appear in this users "to do" list.

Default to Scheme Administrator

The scheme administrator is a nominated individual identified on a group scheme at creation.

 

If ticked, when a workflow item is created automatically for this group by a policy event, diary schedule or as part of a workflow chain, it will appear in the scheme administrators to do" list.

Description

Free form text description of the task.

Tasks on completion

Any tasks that are to be initiated upon completion of this task.

 

This functionality is known as workflow chaining and is often used where a supervisor check forms part of a business process.

Delay

Enables a delay between the closing of a task, and the opening of a chained subsequent task.

 

If left blank, the chained task appears as soon as the predecessor is completed.

Link to function

This is a list representing the servicing functionalities on LIFEfit.

 

If set for the task code, a "start processing" button appears on the workflow task screen which takes the user directly to the relevant servicing screen.

 

This saves time as there is no need to click "maintenance" on the workbench, and find the relevant hyperlink. It also reduces the risk of mistakes.

Unique For Policy

Tick if the task should only be created once for a policy, even if multiple triggers are generated.

 

An example may be if a coverage status change happens on multiple coverages, but only one workflow prompt is required.

Hidden from manual selection

If ticked, the task type is not displayed in the dropdown list when categorising work.

 

This option is ticked for one of two reasons

·         The task should only be created automatically based on a system event (e.g. policy lapse, incoming email)

·         The task is no longer monitored by workflow

 

Prevent manual closing

If ticked, workflow items of this type cannot be closed by a user; they can only be closed by a system event (typically a status change).

 

This ensures integrity in SLA monitoring.

 

e.g. the SLA to process a death claim is 35 hours –workflow is only closed when the policy’s status actually changes.

 

It also simplifies things for the user – there is no need ot separately close workflow as well as complete the work itself.

 

Assign to policy creator

If ticked, when workflow is created automatically it is assigned to the user that created the policy.

 

This is typically used for new business-related tasks.

 

Assign by workload

If ticked, automatically-created tasks of this type are automatically assigned to the least busy person in the default user group (defined above).

 

A user’s “busy-ness” is the sum of target times for items in their to-do list. If they are currently “out of office”, they will not be automatically assigned any new tasks.

 

This feature is particularly useful in high volume processing environments, as there is no need for a supervisor / manager to assign (or users to pick) items from the group to-do list.

 

Event On Creation

Document trigger event on creation of a task of this type.

 

Used to trigger a letter or email.

 

Event On Completion

Document trigger event on closure of a task of this type.

 

Used to trigger a letter or email.
